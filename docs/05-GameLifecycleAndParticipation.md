# Game Lifecycle and Participation

## Goal

Define how a learning game is published, played, managed, and completed.

This document describes the operational flow of a learning game for teachers and students.

---

## Game Lifecycle

A learning game moves through the following states:

### Draft

The teacher creates and edits the learning game.

AI generates the initial content.

Students cannot join.

---

## Lifecycle Actions

Teachers manage learning games through the following actions:

* Publish
* Activate
* Deactivate
* Archive

Publishing prepares the learning game for student participation and moves the learning game to the Inactive state.

Students cannot join until the teacher activates the learning game.

---

### Inactive

The learning game is ready but not available for students.

The teacher can:

* Edit the game
* Change task order
* Update content
* Activate the game

Students cannot join.

---

### Active

The learning game is available for students.

Students can join and play.

The teacher cannot edit the game while it is active.

---

### Archived

The learning game is no longer used.

Students cannot join.

The teacher can view previous results.

Archived games can be reused by changing their state to Inactive.

---

## Student Participation

Students do not need accounts.

To join a learning game, students enter:

* Game Code
* Student Name

After joining, students immediately start playing.
Students can leave the learning game at any time.

If a student leaves the game, the current session ends.

Rejoining starts a new student session.

---

## Learning Game Reuse

A learning game can be used multiple times.

Teachers can:

* Activate a game
* Deactivate a game
* Edit inactive games
* Activate the updated version again

This allows the same learning game to be used by different classes.

If a learning game already contains student results, it cannot be edited directly.

The teacher must create a copy of the learning game before making changes.

This protects existing student results.
---

## Student Progress

Student progress is automatically saved.

If a student leaves the game:

* the current session ends.
* Rejoining starts a new student session.


---

## Game Activation

Teachers control when students can join.

### Active

* New students can join.
* Existing students continue playing.

### Inactive

* New students cannot join.
* Students who already joined can finish their current learning game.

No student should lose progress because a teacher deactivates a game.
While students are finishing an inactive learning game, the teacher cannot edit the game.

Editing becomes available only after all active student sessions are completed or the teacher creates a copy of the learning game.

---

## Teacher Controls

Teachers can:

### Draft or Inactive

* Edit content
* Edit tasks
* Reorder tasks
* Activate the game

### Active

* Monitor participation
* Monitor progress
* Monitor scores
* Deactivate the game

Active games cannot be edited.

---

## Student Session Rules

A student session starts when the student joins the game.

A session ends when:

* All tasks are completed.

Student sessions remain available if the student temporarily leaves the game.

---

## Results

Teachers can review:

* Student Name
* Total Score
* Completed Tasks
* Completion Time

Results become available as students complete their learning games.

---

## Game Lifecycle Principles

* AI creates the first draft.
* Teachers stay in control.
* Students do not need accounts.
* Learning games can be reused.
* Active games cannot be edited.
* Inactive games can be edited.
* Students never lose progress.
* Students should never become permanently blocked.
* Teachers decide when games are available.

---

## Expected Outcome

Teachers can create, activate, deactivate, reuse, and manage learning games with minimal effort.

Students can quickly join, safely continue their progress, and complete learning games without unnecessary interruptions.
